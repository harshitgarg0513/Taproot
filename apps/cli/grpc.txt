import { InterceptingCall } from '@grpc/grpc-js';
import { Inject, Injectable } from '@nestjs/common';
import { REQUEST } from '@nestjs/core';

@Injectable()
export class GrpcClientInterceptor {
  constructor(@Inject(REQUEST) private readonly request: any) {}

  intercept(options, nextCall) {
    const jwtToken = this.request['jwtToken'];
    return new InterceptingCall(nextCall(options), {
      start: (metadata, listener, next) => {
        metadata.add('authorization', `${jwtToken}`);
        next(metadata, listener);
      },
    });
  }
}

To implement a gRPC retry mechanism, we should leverage the `@grpc/grpc-js` library's capability to define a `service_config` with a `retryPolicy`. This is the standard, production-grade way to handle retries in gRPC, as it is offloaded to the gRPC client core rather than being managed manually via custom InterceptingCalls, which are difficult to implement correctly for stateful stream handling.

### Changes Summary

We need to modify the `channelOptions` in your `grpc-client.module.ts` files to include a `service_config`.

#### 1. Modify `apps/apigateway/src/grpc-client/grpc-client.module.ts` (and similarly for `apps/microservice/...`)

```typescript
// ... inside useFactory
options: {
  // ... existing configs
  channelOptions: {
    // 1. Add the retry service config
    'grpc.service_config': JSON.stringify({
      methodConfig: [
        {
          name: [], // Applies to all methods
          retryPolicy: {
            maxAttempts: 3,
            initialBackoff: '0.1s',
            maxBackoff: '1s',
            backoffMultiplier: 2,
            retryableStatusCodes: ['UNAVAILABLE', 'INTERNAL'],
          },
        },
      ],
    }),
    // 2. Keep your existing interceptors
    interceptors: [
      (options, nextCall) => {
        return grpcClientInterceptor.intercept(options, nextCall);
      },
    ],
  },
},
```

### Explanation of Changes

1.  **`grpc.service_config`**: gRPC clients natively support retry policies defined in the service configuration. By setting this in `channelOptions`, you instruct the gRPC core library to automatically retry failed requests based on your criteria without needing custom logic in your `GrpcClientInterceptor`.
2.  **`methodConfig`**: 
    *   `name: []`: An empty list indicates this policy applies to all methods on the channel. You can target specific services or methods (e.g., `[{ "service": "beatroute.dms.ExampleService" }]`) if you prefer granular control.
    *   `maxAttempts`: Total attempts including the initial call.
    *   `initialBackoff`/`maxBackoff`: Controls the exponential backoff timing to prevent overwhelming a service that is just coming back online.
    *   `retryableStatusCodes`: We specifically target `UNAVAILABLE` (common for load balancer transitions or service restarts) and `INTERNAL` (if the service signals a transient internal error). **Note:** Only include idempotent operations (like GET/index/view) if you are worried about side effects on `create` or `delete` requests.

### Important Considerations

*   **Idempotency**: Be cautious. If you retry a `createExample` call that actually reached the server but failed on the response, the server might process the same entity twice. In gRPC, it is highly recommended to implement **idempotency keys** in your protobuf definitions if you enable retries on non-read operations.
*   **Interceptor Scope**: Your `GrpcClientInterceptor` currently injects the `REQUEST` scope. Be aware that if your gRPC call is long-lived or asynchronous, `REQUEST` scope can lead to memory leaks or context loss. The `service_config` approach above is superior because it handles the retry logic at the C++/Core level, ensuring the metadata (Authorization headers) is correctly re-applied on subsequent retries automatically.
*   **Context**: The `apps/microservice` module should be updated using the exact same `channelOptions` pattern to maintain consistency across your microservice fleet.
